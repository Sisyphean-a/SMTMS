# SMTMS 设计哲学与初衷

## 项目缘起

作为一个狂热的《星露谷物语》Mod 玩家，我经常遇到一个问题：安装的大量 Mod 几乎都是英文名称和描述，这就导致：

1.  **难以区分**：在成百上千个 Mod 中寻找特定的功能模组非常困难。
2.  **管理混乱**：手动管理这些 Mod 的分类和更新极其繁琐。

为了解决这个问题，我最初的做法是手动修改 `manifest.json` 文件，将 Mod 的 `Name` 和 `Description` 翻译成中文。这种做法虽然有效，但带来了新的痛点。

## 原有痛点分析

在开发 SMTMS 之前，我尝试使用 Python 脚本来辅助这一过程，但存在以下严重缺陷：

1.  **缺乏可视化 (UI)**：只能通过命令行运行脚本，不直观，无法即时预览修改结果。
2.  **安全隐患**：
    - **误操作风险**：总是担心跑错脚本，导致辛苦翻译的中文被英文原版覆盖，或者反之。
    - **缺乏版本控制**：一旦覆盖错误，无法一键回滚。
    - **格式破坏**：简单的 JSON 解析库往往会移除 `manifest.json` 中的注释（C# 风格的 `//` 或 `/* */`），而这些注释往往包含重要信息。
3.  **操作繁琐**：
    - 需要手动复制 Nexus Mods 的 URL 页面。
    - 修改名称后，需要手动维护分组文件。
4.  **翻译效率低**：很多 Mod 只需要简单的机翻即可通过，人工翻译极其耗时。

## 核心设计理念

为了彻底解决上述痛点，SMTMS 应运而生。其核心设计理念如下：

### 1. 安全至上 (Safety First)

- **Git 集成**：翻译数据库在 Git 监控下，任何误操作都可以回滚。Git 仓库位于工具自己的数据目录，**不在 Mods 目录创建任何文件**。
- **非破坏性编辑**：在应用翻译到 `manifest.json` 时，必须使用**正则表达式 (Regex)** 而非简单的 JSON 序列化，以完美保留原始文件中的注释和特殊格式。
- **自动化版本控制**：Git 操作完全自动化，用户无需理解 Git 概念，系统在后台自动提交变更。

### 2. 可视化与直观 (Visual & Intuitive)

- 告别命令行，拥抱现代化 UI。
- 用户应该能清楚地看到哪些 Mod 已翻译，哪些未翻译，哪些已应用。
- 所有的操作都应通过简单的点击完成，无需多步骤流程。
- 历史查看直观，回滚操作简单明了。

### 3. 先进的自动化 (Advanced Automation)

- **数据库化管理**：用数据库完全替代 `xlgChineseBack.json`，支持高效查询、分组和统计。
- **AI 辅助**：未来应集成 API（如 DeepL、ChatGPT），实现一键自动翻译 Mod 名称和描述，解放双手。
- **智能缓存**：AI 翻译结果缓存到数据库，避免重复翻译相同内容。

### 4. 非侵入性 (Non-Invasive)

- **不在 Mods 目录创建任何文件**：工具数据存储在 `%APPDATA%\Local\SMTMS\`，与用户现有工作流零冲突。
- **只读访问 Mods 目录**：仅在扫描时读取 `manifest.json`，在应用翻译时写入，其他时间不干扰。
- **不与现有 Git 仓库冲突**：如果用户已在 Mods 目录使用 Git，本工具不会产生任何影响。

### 5. 专注核心 (Focus on Core)

- **定位明确**：这是一个**翻译管理工具**，不是全功能 Mod 管理器。
- **功能聚焦**：专注于 `Name` 和 `Description` 的中英文对照管理。
- **简单易用**：避免功能蔓延，保持工具简单直观。

## 总结

SMTMS 是一个为重度 Mod 玩家打造的**简单、安全、高效**的翻译管理工具。任何后续功能的开发，都不能背离以下三个基本点：

1. **数据安全**：翻译数据有版本控制，可随时回滚
2. **用户体验**：操作简单直观，无需学习成本
3. **非侵入性**：不干扰用户现有工作流，不在 Mods 目录创建文件
